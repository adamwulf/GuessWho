## Architecture (converged with the three planning-consult agents; approved by two)

Two-process **relay**. The running **Mac Catalyst app** is the HOST: it owns TCC grants (EventKit/Contacts), the single `GuessWhoSync`/`ContactsRepository` instance, the iCloud sidecar, and all human-confirmation UI. A thin **relay** binary (`guesswho`) — bundled at `GuessWho.app/Contents/MacOS/guesswho`, optionally symlinked to `/usr/local/bin` — is BOTH front doors:

- `guesswho run` → long-lived **MCP stdio server** (for Claude Desktop / Cursor / Claude Code).
- `guesswho contacts list …` (etc.) → one-shot **human CLI**.

Both send the same typed requests over **App-Group named pipes** to the app; the app runs the logic and answers. The relay **does not link `GuessWhoSync`** — only the shared wire module + transport package.

```
Claude Desktop / terminal
   │ MCP JSON-RPC over stdio   |  ArgumentParser CLI flags
   ▼
 guesswho (relay; bundled in .app; NOT linked against GuessWhoSync)
   │ newline-delimited JSON over named FIFOs (App Group container)
   ▼
 GuessWho.app (host — owns TCC, the single GuessWhoSync, the sidecar, confirm UI)
   │ direct calls into the SAME ContactsRepository/SyncService the UI uses
   ▼
 EventKit / Contacts + iCloud JSON sidecar
```

### Why relay, not a direct-to-Sync standalone CLI
1. **TCC ownership** — EventKit/Contacts auth is per-process and needs UI to grant; a client-spawned headless relay can't cleanly hold it. The running app already has it.
2. **Single writer** — `GuessWhoSync`'s per-key `NSLock` tables + contact-change watcher are per-process. Sidecar *files* are `NSFileCoordinator`-safe (no corruption), but two owners could still interleave read-modify-write and bypass the in-memory locks. One owner = correctness for free.

### v1 SCOPE STATEMENT (prevents scope creep — write this into the plan verbatim)
**v1 mutates GuessWho *sidecar* data only; system-store (EventKit / Contacts) *content* is READ-ONLY.** Reading system contacts/events is in; writing them is out of v1. Contact/event content writes are a *different coordination domain* (EventKit/Contacts, not the relay's single-writer sidecar guarantee), they need TCC *write* grants (a strictly harder grant than read with no clean mid-flight rollback), and they'd entangle the legitimate write path with the still-pending `com.apple.developer.contacts.notes` entitlement — so they get their own design pass, not a v1 phase.

### Load-bearing invariants (acceptance criteria, not conventions)
- **INV-1 Single writer by construction:** the relay target does NOT link `GuessWhoSync`. Enforced by the link line, verified in the build.
- **INV-2 Writes are visible everywhere:** every write tool routes through the SAME `ContactsRepository`/`SyncService` entry points the UI uses, so the change-watcher, iCloud push, and UI observation all fire. Test: an MCP-driven write is visible in the running app's UI and pushed to iCloud **without relaunch**.
- **INV-2b Host uses the LIVE app instance (read-side mirror of INV-1):** the host holds a reference to the app's existing `ContactsRepository`/`SyncService` (injected at app launch) and NEVER constructs its own store. **Enforcement = ARCHITECTURE CHECK, not a unit gate** (corrected after review): `ContactsRepository` is a *write-through read-model cache* (`ContactsRepository.swift:26` "not a second source of truth") — writes persist straight through, so there is NO tool-visible "un-persisted in-memory delta" to observe, and a naive positive-consequence test has no nameable state. Instead: a **link-layer / grep check** that the relay + host module never call a `ContactsRepository(...)` or `GuessWhoSync(...)` constructor (the instance is injected). **INV-2 is the behavioral proof** that the shared-instance wiring is correct (an MCP write shows up in the live UI + iCloud without relaunch — the consequence we actually care about, in the testable direction).
- **INV-3 Apple note NEVER crosses the wire (either direction):** the wire contact DTO has **no** `note`/`notes` field; the host mapper never reads `Contact.note` (Apple); no write tool accepts a note-shaped param; **search never MATCHES the Apple note** (a search that matches note text but returns note-less DTOs still leaks note content via *match-presence* — an agent binary-searches queries to reconstruct the note; this is a distinct assertion from "note absent from output"). The **allowed** "note" on the wire is GuessWho's own sidecar `ContactNote` (via `notes(at:)`), a *different type*. Pin the host search to `Contact.matches(searchQuery:)` (verified note-free, `Contact+Search.swift:63`) so the invariant doesn't rest on an unwritten implementation choice. **Tests (adversarial, structural):**
  - *Output absence:* a fixture whose Apple `Contact.note` holds a unique **sentinel** — the sentinel appears in **zero** tool outputs across read/list/search/write-echo **AND error responses** (force a failing call against the fixture, scan the error JSON); assert the key is **absent** from the encoded JSON (not merely empty).
  - *Search oracle:* `contacts.search(query: <note-sentinel>)` returns **ZERO** results (and for substrings of the sentinel) — proves match-presence can't leak note content.
- **INV-3b No internal-identity leak via URLs/search:** `Contact.urlAddresses` includes the package-owned `guesswho://contact/<uuid>` identity URL; **only `Contact.userVisibleURLAddresses` (`Contact.swift:177`) filters it** — the wire DTO MUST source URL addresses from `userVisibleURLAddresses`, never raw `urlAddresses`. Also, `Contact.matches(searchQuery:)` iterates **raw** `urlAddresses` (`Contact+Search.swift:86`), so `search("guesswho")` matches every reconciled contact — pinning search to `matches()` does NOT close this; the wire search path must pre-filter identity URLs (or the package must match on `userVisibleURLAddresses`). **Test:** `contacts.search(query: "guesswho")` returns ZERO matches, and `guesswho://` appears in zero tool outputs (part of the banned-vocabulary test below).
- **INV-4 App + helper derive the app-group id from ONE shared per-channel build var (never a hardcoded string):** ✓VERIFIED in Muse — the group id is team-prefixed AND channel-suffixed (`group.<TEAM>.<parent-bundle-id>`), so the *string differs per channel*. **THREE distinct id derivations** (App Store, Setapp, Debug), across four nominal channels (TestFlight rides the App Store id). Debug matters: GuessWho already uses `.debug`-suffixed bundle/App-Group/iCloud ids per [[project_debug_build_identifiers]] — the per-channel derivation MUST include Debug or Debug builds silently mismatch (app + helper resolve to different containers). It works only because both targets expand the group from the same build variable within a build. A hardcoded constant, or two vars that drift, = silent FIFO failure. INV-4 is the *static* guarantee; **Phase 0 exit criterion 3 (app + CLI resolve the SAME container path at runtime, on each channel's build INCLUDING Debug) is its runtime verification** — named together on purpose.
- **INV-5 Mac (Catalyst) only for v1:** pipe/wire code gated Mac-only so iOS builds don't pull in FIFO code with no host. Concretely: the relay binary is built + Copy-Files-embedded ONLY on the Catalyst-capable config; **an iOS-only build simply OMITS the Copy Files phase** (it must NOT try to embed a macOS Mach-O into an iOS `.app` — that would fail to codesign). Make the iOS build's absence-of-relay explicit so nobody wonders where it went.

### Consent & safety model (NO per-call confirmation dialogs)
The MCP client is headless — the human is in Claude Desktop / Terminal, not staring at GuessWho — so an in-app modal on an MCP write is a dialog **nobody is watching** (it blocks/times-out the call or forces alt-tab per write). Therefore: **consent is the read-only toggle, granted once in Preferences** (`isMCPReadOnly` / `isCLIReadOnly` — writes OFF by default). Destructive ops are safe because GuessWho deletes are already **soft-delete/tombstone** (`deleteNote` sets `deletedAt`; `notes(at:)` filters live, `allNotes(at:)` recovers) — recoverable, so no hard-delete in v1 and no per-call prompt needed. This also removes the "longer interactive timeout" complexity. (Note: the inherited `ResponseManager` default is **5s**, `ResponseManager.swift:110` — not 10s; GuessWho sets timeouts **per-tool in metadata** regardless, so the exact inherited default is moot.) If Adam wants human-in-the-loop approval for agent writes, design it out-of-band (a notification the user acts on), NOT a modal blocking the tool call — flag as a product decision.

## Phase 1 — Transport spine + read-only tools end-to-end

**Goal:** the full spine — relay ⇄ pipes ⇄ host dispatch ⇄ real data ⇄ wire DTO — proven with **read-only** tools. This is where INV-2/3/4 first get tests.

**Work:**
- Vendor `mcp-template`'s EasyMCP + EasyMacMCP (or the pieces we need) as a dependency. Respect the pipe gotchas: reader **keepalive FD** (avoid EOF busy-spin); shutdown ordering (cancel task → wake sentinel `\n` → await → close); persistent `AsyncLineSequence`; one request = one newline-delimited JSON line; helperId + messageId on every message.
- **TRANSPORT CHANGE — per-helper REQUEST pipe (locked in):** EasyMacMCP today uses ONE shared central request FIFO (many helpers → one host) and raw newline-JSON (NOT length-framed). On a shared FIFO, only writes ≤ **PIPE_BUF (512 bytes on Darwin — NOT 64KB; the 64KB figure was wrong)** are guaranteed atomic; above that, two concurrent helpers' writes can INTERLEAVE and tear the newline-framed stream. GuessWho requests can carry large notes/markdown (far over 512B), so the shared-pipe design is a latent tearing bug under concurrent clients. **Fix by mirroring the existing per-helper *response* pipe on the request side:** one request FIFO per `helperId` → exactly ONE writer per pipe → interleaving is structurally impossible at ANY size (a 500KB note writes in multiple non-atomic chunks nobody interleaves with; the line reader reassembles). Keeps newline-JSON framing UNCHANGED (no length-prefix parser on either side), symmetric with `[helperId: HostResponsePipe]` (add `[helperId: HostRequestPipe]`).
- **DISCOVERY (the one cost of per-helper request pipes — nail in Phase 1):** unlike the central request pipe (fixed path, app pre-opens it), the app must LEARN each new helper's request-pipe path. Design: a small **central announce channel** (fixed path) carries ONLY tiny `initialize(helperId:)` / `deinitialize(helperId:)` control messages — these are well under 512B so the shared-pipe atomicity ceiling is never a problem for them. On `initialize`, the app spins up that helper's dedicated request-reader + response-writer, then sends `ready` (Guard 2); on clean `deinitialize` it tears them down. This reuses the helperId routing key already present everywhere. **The ENXIO-fast-fail open probe is SCOPED to the ANNOUNCE channel ONLY** (corrected after review): ENXIO on `O_WRONLY|O_NONBLOCK` open means "app not running" *only* on a pipe the app pre-opens at launch — that's the announce/central channel (host opens its reader in `startListening`), so the probe is the app-liveness signal THERE. It is NOT valid on `request_pipe_<helperId>`, where ENXIO/`pipeDoesNotExist` during the post-`initialize` window is a *transient* "host hasn't finished `setupHelperPipes` yet" while the app IS running — a uniform fast-fail would spuriously report "app not running" on the first tool call. The per-helper request pipe is opened only AFTER the ready-ack, at which point any subsequent ENXIO/EPIPE unambiguously means "app died mid-session" → disconnect + re-announce (see reaping/reconnect). Keep the per-helper RESPONSE pipe + messageId matching regardless.
  - **DESIGN RULE — the announce channel keeps the 512B constraint forever:** it is still a many-writers→one-reader shared FIFO; it is safe ONLY because init/deinit are tiny. **No field of `initialize`/`deinitialize` may ever grow unboundedly** — do NOT add a "client capabilities/metadata" blob to `initialize` in a later version, or the tearing bug re-arms on the control channel. State this in the wire-module doc, not just as the Guard-1 assert.
- **DEAD-HELPER REAPING (Phase 1, NOT Phase 3 — the teardown code is written here):** a helper killed by SIGKILL / crash / client force-quit NEVER sends `deinitialize`. Two consequences, both worse under per-helper request pipes: (a) the host's per-helper request-reader Task + FDs leak during ONE long app run, and (b) `request_pipe_<helperId>` + `response_pipe_<helperId>` FIFO files leak in the group container and accumulate across runs. **Critically, reader EOF is NOT a usable death signal:** `ReadPipe` deliberately holds its OWN `O_WRONLY` keepalive FD (the busy-spin fix — `ReadPipe.swift:31-37`, opened `:151-169`) so an external writer detaching does NOT deliver EOF; a dead helper leaves the per-helper request reader parked, not EOF'd.
  - **LIVENESS IS REQUIRED (not one-of-two — corrected after review).** A launch-sweep alone fixes ONLY consequence (b); it does NOTHING for (a), the leaked reader Task + FDs during one long run — which is exactly the `kill -9` case this motivates. So: a **last-seen timestamp per helperId** (bumped on any request + a periodic helper ping); on timeout the host runs the SAME awaited-per-pipe teardown as a clean `deinitialize`.
  - **Launch-sweep is OPTIONAL belt-and-braces** for stale FIFO files across runs — but it must NOT assume "a fresh app owns no helpers." That premise is FALSE: an MCP client keeps `guesswho run` ALIVE across app relaunches, so a naive sweep deletes a LIVE helper's FIFOs and silently wedges it. The sweep is safe ONLY in combination with helper re-announce (below): a live helper re-registers and re-creates its pipes, so sweeping orphans is harmless.
- **HELPER RECONNECT AFTER HOST RESTART (Phase 1 — currently undesigned, REQUIRED):** the helper's response `ReadPipe` also holds its OWN keepalive writer FD (`ReadPipe.swift:31-37`), so an app quit delivers NO signal to a parked helper — it discovers host death only via EPIPE/write-failure on its next request. Design: on write failure (or ready-timeout), the helper **closes both its pipes, re-runs the announce-channel liveness probe** (ENXIO ⇒ surface "GuessWho is not running" upward / poll), then **re-sends `initialize` and awaits `ready`** (Guard 2's handshake makes this loop clean) before retrying. This makes the launch-sweep safe (live helpers re-register after a relaunch).
- This shapes the discovery/teardown design, so decide it BEFORE writing the teardown code (retrofitting after discovery is built = churn). Looks fine in dev (Ctrl-C → deinit fires) and leaks/wedges in the field (clients `kill -9` helpers, and users quit+reopen the app under a live client).
  - **Guard 1 — channels strictly typed apart:** the announce channel carries ONLY init/deinit, NEVER a real request. Add a debug assertion that any frame written to the announce pipe is **≤ PIPE_BUF (512B on Darwin)** — POSIX guarantees atomicity for writes ≤ PIPE_BUF. The moment a real (large) request is routed through the announce channel "just once," the tearing bug returns on the control channel.
  - **Guard 2 — open-before-first-request ordering (a discovery-race that tears under load) — REQUIRES a ready-ack handshake (corrected after review; the earlier "solved for free" claim was WRONG for the per-helper topology).** Inline lifecycle dispatch (`HostRequestPipe.startReading` handles `isInitialize`/`isDeinitialize` inline, `HostRequestPipe.swift:62-76`) only orders messages arriving on the SAME pipe. In Muse's central-pipe design that suffices because init AND tool calls share one FIFO. **Our per-helper split breaks it:** tool calls move to `request_pipe_<helperId>`, a FIFO the HOST creates+opens only when it processes `initialize`. The race moves to the HELPER side — the helper's `WritePipe.open` on `request_pipe_<helperId>` can hit `pipeDoesNotExist` (`WritePipe.swift:86-88`, before the host's `mkfifo`) or ENXIO `openFailed` (`WritePipe.swift:97-101`, FIFO exists but host reader not yet open). The window is real: MCP clients send `initialize`→`listTools` back-to-back. **Fix — promote the ready-ack from "fallback" to the DESIGN:** the helper already creates+opens its response `ReadPipe` BEFORE sending `initialize` (`MCPHelper.swift:90-94`), so at the end of `setupHelperPipes(for:)` the host writes a tiny **`ready`** control message on that helper's RESPONSE pipe. **`ready` MUST echo the `initialize` request's `(helperId, messageId)`** so the helper awaits it through the existing `ResponseManager.waitForResponse(helperId:messageId:)` path — that's what makes it "zero new receive-side plumbing"; an unsolicited fresh-messageId control message would instead need a new ResponseManager hook. The helper AWAITS `ready` before it opens `request_pipe_<helperId>` / sends its first tool call. (The inherited `waitForResponse` 5s default becomes the ready-timeout unless overridden.) Treat **ready-timeout** as a distinct error state ("app running but setup failed") — NOT the same as announce-channel ENXIO ("app not running") — so both failure modes stay diagnosable. This IS new code; the plan no longer claims otherwise.

- **IMPLEMENTATION ANCHORS (inherit, don't invent — source-traced by essentialmcp):**
  - Discovery is **message-driven off `initialize`**, NOT dir-watching (dir-watching is fragile: FSEvents latency, partial-create races, stale-FIFO cleanup). Templates: `EasyMCPHost.setupResponsePipe(for:)` (EasyMCPHost.swift:79) + `HostRequestPipe.startReading` (HostRequestPipe.swift:62).
  - The **only** new code vs today: keep the central `HostRequestPipe` as an **announce-only** reader (it only ever sees init/deinit, which already dispatch inline), and extend `setupResponsePipe` → `setupHelperPipes(for:)` to ALSO create a per-helper `HostRequestPipe` on `request_pipe_<helperId>` (opened + `startReading{ handleRequest }`), stored in `[helperId: HostRequestPipe]` alongside the existing `[helperId: HostResponsePipe]`. `teardownHelperPipes` closes BOTH — invoked on clean `deinitialize` OR by the reaper (see DEAD-HELPER REAPING; do NOT rely on reader EOF, the keepalive FD suppresses it). Helper side: init/deinit → announce FIFO; all tool calls → `request_pipe_<helperId>`.
  - **BUG TO FIX ON INHERIT:** EasyMacMCP's `WritePipe.write` doc comment (`WritePipe.swift:131`) claims "PIPE_BUF is 65536 on Darwin/Linux" — **wrong for BOTH OSes.** Darwin `PIPE_BUF` is **512** (`sys/syslimits.h:104`); Linux `PIPE_BUF` is **4096** (65536 is Linux's default pipe *capacity* via `F_SETPIPE_SZ`, a different thing). Fix the comment to the correct values. (This is also why per-helper request pipes are correctness, not just large-note support: even one contact JSON can exceed 512B and interleave on a shared pipe.)
  - **BUG NOT TO COPY (worse than first stated — verified):** the base `EasyMCPHost.stopListening` (`EasyMCPHost.swift:57-71`) closes pipes in **un-awaited detached `Task {}`s** — including the central REQUEST pipe, itself a deadlock-sensitive READ pipe (`:59-61`), not just the response write pipes. Tolerable for write pipes, **WRONG for the read pipes**. Our per-helper request READERS (and the announce reader) must have their `close()` (which runs cancel → `signalReaderWake` → await task → `readPipe.close()`, `HostRequestPipe.swift:44/:95`) **awaited individually** on teardown — do NOT fire-and-forget in a loop.
- **Shutdown bite:** the host now runs N request-reader tasks; each must follow cancel → `signalReaderWake` → await → close PER pipe, or the dispatch_io deadlock reappears N times. Keep a ≤PIPE_BUF **tripwire assert** on the announce channel only (control messages must stay tiny); per-helper data pipes need no size cap.
- **Shared wire module** (compiled into BOTH app + relay): `MCPRequest`/`MCPResponse` enums (one case per tool), note-**less** contact/event DTOs, `create(helperId:messageId:parameters:)` validation, pipe-path constants (from Info.plist / xcconfig), the `ready(helperId:)` control message, and a **typed error enum** (below).
- **Wire IDENTITY semantics (specify explicitly — from review):** contact IDs on the wire are the **opaque `ContactID` token**, NEVER `localID` — per `docs/contact-identity.md`, `localID` must never be persisted/compared, and an MCP agent caching an ID across a conversation IS persistence. Document the caveat that `ContactID` falls back to `localID` **pre-reconciliation**, so a wire ID can be unstable until the contact reconciles. **Events:** expose the **stable sidecar UUID when the event has one**, else an opaque `eventKitID`-derived token; reads do **NOT mint sidecars** (match the no-mint semantics of `fields(for:)` — adoption/mint stays a UI path, never an MCP-read side effect). **Decision (made now): `events.list` DOES surface un-adopted EventKit-only events** (they're real events the user sees in the app), each carrying the `eventKitID`-derived opaque token as its wire ID; a sidecar UUID appears only once an event has been adopted through the UI — the MCP read never triggers adoption.
- **DTO field sourcing (leak-safe):** the contact DTO sources URL addresses from **`userVisibleURLAddresses`** (`Contact.swift:177`), never raw `urlAddresses` (which carries the `guesswho://` identity URL — INV-3b). `Event.eventKitNotes` (`Event.swift:22`) MAY ride the event DTO — it is the **Calendar event note, NOT the Apple Contacts note**, so it is ALLOWED; state this explicitly so nobody "fixes" it in either direction.
- **Typed error taxonomy (wire module) + PER-CALL enforcement:** a typed error enum — `notRunning`, `disabled`, `permissionDenied`, `readOnly`, `tooLarge`, `notFound`, `invalidParams`. Error payloads carry **typed codes + plain strings only — NEVER a serialized model type** (a `Codable` `Contact`/`Event` in an error message would leak the Apple note; forbid interpolating `Contact`/`Event` values into any wire string). The master/read-only/permission gates are enforced **PER-CALL server-side**, not merely by hiding tools from `listTools` — an MCP client CAN call an unlisted tool, so hiding is UX, not enforcement.
- **Host** (`GuessWhoMCPHost: EasyMCPHost` in the app): `handleRequest` switch → per-tool handlers calling the SAME `ContactsRepository`/`SyncService` the UI uses. The host is **injected** with the app's live instance at launch (INV-2b is an architecture-check, not a unit gate). `contacts.search` is pinned to `Contact.matches(searchQuery:)` with an identity-URL pre-filter (INV-3 / INV-3b). Dynamic, permission-gated `listTools`; but gates are ALSO enforced per-call (see error taxonomy).
- **DISPATCH MODEL (Phase 1 — from review, avoids re-creating the 2026-07-02 hang):** `ContactsRepository` is `@MainActor` (`ContactsRepository.swift:31`), so routing tool calls through the live instance puts work on the UI main actor. **Hop to `@MainActor` ONLY for the repository read/write; do the JSON encode + pipe write OFF the main actor.** Otherwise agent bursts compete with the UI, a busy main thread (reconcile, photo import) eats the response timeout, and encoding large responses on main recreates the main-actor-I/O hang class ([[project_mainactor_io_offload]]).
- **NO-HOST behavior (the most common real state — client connects while the app isn't running):** MCP clients call `tools/list` at connect. Define it: return an **empty tool list + a plain-language status** ("GuessWho is not running — open the app"), and send `notifications/tools/list_changed` after the helper re-handshakes (reconnect design) so the client re-queries. Never leave it undefined.
- **Relay CLI**: `run` (MCP stdio) + read-only subcommands.
- Read-only tools v1: `contacts.search`, `contacts.get`, `contacts.listNotes` (sidecar notes), `contacts.listTags`, `contacts.listCustomFields` (via `ContactsRepository.fields(for:)` so the `.blob` filter is inherited — Phase 2 adds writes; read + write should land together, no write-without-read asymmetry), `contacts.listRelationships` (contact↔contact links — wire name avoids "link"; see banned-vocabulary note), `contacts.listFavorites`, `events.list`, `events.get`, `guides.list`, `guides.get`, `places.list`. `events.list` takes an explicit **date-window** param (start/end); the attendee-based recent-events surface is intentionally OUT of v1 (state it, don't leave it implied).
- **Master toggles** in App-Group UserDefaults: `isMCPEnabled` / `isCLIEnabled` (OFF by default; user opts in via app Preferences) + `isMCPReadOnly` / `isCLIReadOnly` (server-side enforced).
- **Bounded reads from day one:** every list/search read takes a `limit` param (+ a cursor/offset for pagination) and enforces a max **response** size. **Decision (made now, per review): a page that would exceed the cap returns the typed `tooLarge` error with guidance to narrow the query / lower `limit`** — NOT a silent truncation (a truncated list read as complete is a correctness trap). Pagination via the cursor is the sanctioned way to get the rest. Retrofitting pagination after clients depend on unbounded lists is painful; cheap now, expensive later. (Request side is handled by the per-helper request pipe above; this is the response side.)

**Exit criteria:**
- End-to-end: an MCP client lists + calls a read tool against the running app and gets real data.
- **INV-3 test lands here (first tool):** (a) *output absence* — Apple-note sentinel appears in zero tool outputs across read/list/search/write-echo AND error responses; structural key-absence in encoded JSON. (b) *search oracle* — `contacts.search(<note-sentinel>)` returns ZERO results (+ substrings).
- **INV-3b test:** `contacts.search("guesswho")` returns ZERO matches; a fixture carrying a `guesswho://` identity URL exposes it in zero tool outputs (URL DTO sourced from `userVisibleURLAddresses`).
- **INV-1 test (CI gate):** relay binary does not link `GuessWhoSync` (build/link-line check).
- **INV-2b (architecture check, not a unit gate):** a grep/link-layer check that the relay + host module never call a `ContactsRepository(...)`/`GuessWhoSync(...)` constructor (instance is injected). Behavioral proof is INV-2 (Phase 2).
- **INV-5 test (CI):** an iOS build of the app target compiles green with the relay + pipe/wire code excluded (catches the day someone `#if`s the gating wrong).
- **Banned-vocabulary test (enforces the product principle — was asserted, now tested):** serialize the FULL `listTools` output (tool names + descriptions + param docs) AND all error/status strings and assert NONE contain the mechanical-grep terms: `sidecar`, `unlink`, `EventKit`, `Calendar event`, `reconcile`, `guesswho://`. The word **"GuessWho" needs an allowlist**, because the plan itself REQUIRES the product name in a status string ("GuessWho is not running — open the app") — so the test bans "GuessWho" EXCEPT within a small set of known legitimate product-name phrases, and the "GuessWho used as a noun for our *private records*" concern is otherwise a review-checklist judgment call, not a grep.
- **Transport test — concurrent large request (proves the headline fix):** a request carrying a >512B payload (e.g. a multi-KB markdown note body) round-trips intact, AND two concurrent helpers each sending >512B requests both arrive uncorrupted. Without this, the per-helper-request-pipe change is unverified — the tearing bug is silent, load-dependent, and passes casual testing.
- **Transport test — rapid teardown (proves the await-each-close fix):** rapid connect / disconnect / reconnect of a helper N times leaks no reader Tasks and never wedges (guards the un-awaited-detached-close bug we are NOT copying).
- **Transport test — HOST restart under a live helper (proves reconnect + reaping):** kill + relaunch the APP while a helper stays alive; the helper's next tool call succeeds after an automatic re-handshake (re-`initialize` + `ready`), and the launch-sweep does not wedge the live helper. (The rapid-teardown test covers helper churn only; this covers host churn.)
- **No-host test:** with the app not running, an MCP client's `tools/list` returns an empty list + the plain "GuessWho is not running" status (no hang, no error spew).
- Tools hidden when permission not granted or master toggle off — AND the same gate rejects a direct call to an unlisted tool per-call (hiding is UX, enforcement is server-side).
- Plain-language schemas only — verified by the banned-vocabulary test above, not just asserted.

## Phase 2 — Write tools (GuessWho sidecar data only)

**Goal:** mutations over GuessWho-owned data, routed through the app's real write paths. **No per-call confirmation dialogs** — consent is the read-only toggle (see model above); destructive ops are safe because GuessWho already soft-deletes (tombstones).

**Work:**
- Write tools over **GuessWho-owned sidecar** data only: add/edit/delete **note** (`ContactNote`), add/edit/delete **tag**, add/remove **relationship** (contact↔contact link — wire name is "relationship", not "link"; "link" is on the CLAUDE.md suspect-word list), set/clear **favorite**, add/edit/delete **custom field**, guide/place create/reorder/delete. ("**sidecar custom field**" is INTERNAL naming only — the wire tool name + description say plain "custom field", per the Phase 1 plain-language ban.) (Per v1 SCOPE STATEMENT, EventKit/Contacts *content* writes are OUT of v1 — their own design pass.)
- **CUSTOM-FIELD WRITE GUARDRAILS (from review — the `.blob`/`previousPhoto` clobber risk):** the app UI already learned this — `ContactsRepository.fields(for:)` (`:1161`) EXCLUDES `.blob` fields (the previousPhoto phantom-row bug, [[project_blob_fields_leak_into_custom_fields_ui]]), and the upsert-by-name path (`:1167`) REPLACES an existing field of a DIFFERENT type. So the write tool MUST: (a) allow only wire-writable field types — **text / note / multilineNote / date / checkbox; `.blob` is rejected** (else an agent injects blob pointers); (b) **reject writes to reserved internal names** — centralize the reserved list next to `previousPhotoFieldName` (`ContactsRepository.swift:540`); an agent writing a field named `previousPhoto` would clobber the photo-restore snapshot via the type-replace upsert; (c) the read side (`contacts.listCustomFields` / `contacts.get` fields) goes through `fields(for:)` so the `.blob` filter is inherited. **Tests:** a `.blob`-typed write AND a write to a reserved name (e.g. `previousPhoto`) are BOTH rejected.
- **Delete = soft-delete/tombstone only** (verified across every listed surface — notes, fields, links/relationships, guides, places all tombstone + support undelete; favorite is a reversible toggle). No hard-delete in v1.
- **Read-only gate** (`isMCPReadOnly` / `isCLIReadOnly`) rejects writes **per-call server-side** (not merely by hiding from `listTools`); writes OFF by default. This IS the consent gate — no modal.
- Standard response timeout holds (no interactive dialogs to wait on). Timeout stays **per-tool declarative** in tool metadata so a future interactive tool can opt into a longer window without a global change.

**Exit criteria:**
- **INV-2 test:** an MCP-driven note/tag write is visible in the running app's UI and pushed to iCloud **without relaunch**.
- **INV-3 write-direction test:** no write tool accepts a note-shaped param that could set/clear the Apple `Contact.note`.
- Writes rejected when `isMCPReadOnly`/`isCLIReadOnly` is on; a "deleted" item is recoverable (tombstone, not destroyed).

**Review cycle** → then Phase 3.

## Phase 3 — Preferences UI, install UX, docs, packaging polish

**Goal:** ship-ready.

**Work:**
- App **Preferences**: enable MCP / enable CLI toggles, read-only toggles, **copy-helper-path button** (PRIMARY install on every channel), install/repair `/usr/local/bin` symlink (auth panel, best-effort where the sandbox permits), and guidance to remove the symlink (`rm` — no authorized-delete API).
- **Copy-path is the whole "install" on copy-path channels ✓VERIFIED (Muse):** the button action is literally `UIPasteboard.general.string = helperPath` (from the single `Bundle.main.url(forAuxiliaryExecutable:)` helper). No generated `.mcp.json`, no writing to the client's config. The user pastes the absolute path into their client's `command` field; for us the config is `{"mcpServers":{"guesswho":{"command":"<pasted path>","args":["run"]}}}`.
- **Stale-symlink / resolve-back-to-bundle — the 4-state resolver ✓VERIFIED (Muse `CLISymlinkResolver`, ~60 lines, pure + unit-testable; copy near-verbatim):**
  - **Discriminator:** `try? FileManager.destinationOfSymbolicLink(atPath:)` is the AUTHORITATIVE is-symlink check — it succeeds even on a DANGLING link where `fileExists` lies. Do NOT use `attributesOfItem`/`fileExists` alone; they follow the link and can't tell the four states apart.
  - **Four states:** `notInstalled` (no symlink, no file) · `conflictingFile` (a REAL file/dir occupies the path — tell the user to `rm` it, don't clobber) · `dangling` (symlink whose destination is gone — offer "Reinstall to repair") · `installed` (symlink whose destination exists AND resolves to OUR bundle).
  - **Bundle-identity check (catches the multi-install / wrong-channel trap):** compare `URL(…).resolvingSymlinksInPath().path` of BOTH the symlink destination AND the expected bundle target before declaring `installed`. `resolvingSymlinksInPath` on **both** sides is load-bearing — Setapp/Sparkle bundle paths contain symlinks themselves, so a raw string compare false-negatives a legit install. If resolved-dest ≠ resolved-expected → a symlink pointing at a DIFFERENT (older/other-channel) bundle → treat as **conflict**, not installed. This is exactly why per-channel matters: a Setapp symlink pointing at the MAS bundle surfaces here as a conflict instead of silently talking to the wrong app. Keep wire changes **additive-only** (the only wire-skew surface).
- **`.mcp.json` absolute-path repair:** the client config hardcodes `.../Contents/MacOS/guesswho`; a MAS in-place update or a user moving the app can invalidate it, and the relay then silently stops resolving (same skew surface as the symlink, but for the direct-path install). On app launch, verify the shipped helper path resolves; surface a repair hint in Preferences if a known client config points at a stale path.
- `docs/cli-mcp.md`: architecture, the two-process model, tool list, the Apple-note exclusion invariant, install steps, `.mcp.json` example.
- **App-Review readiness for TCC-Contacts egress (the re-opened risk):** review + update the `NSContactsUsageDescription` purpose string and the privacy nutrition label to reflect that a user-enabled local agent interface reads contact data; prepare **App Review notes** stating the mitigations — local-only IPC (App Group FIFOs, no network), data never leaves the machine, MCP/CLI toggles OFF by default + explicit user opt-in, read-only default. Coordinate timing with the pending contacts-notes entitlement request (54G9JZR59F).
- CI: build the helper target; run the wire round-trip + note-exclusion + banned-vocabulary + INV-1/INV-2b(arch-check) tests.

**Exit criteria:** a user can enable MCP from Preferences, wire up a client, and use read + GuessWho-owned-write tools, with the Apple note provably never exposed.

**Review cycle** → done (v1).

## Design notes carried from the planning session
- **Constants:** no second code-gen (EssentialMCP's BuildSettings.swift + assume-unchanged is a per-clone team papercut). Fold pipe/app-group/bundle-id constants into GuessWho's **existing xcconfig** pipeline → Info.plist → `Bundle.main.object(forInfoDictionaryKey:)`. App-group id **per-channel-derived** from ONE shared build var referenced by both targets (NOT a hardcoded string — it differs App Store vs Setapp; INV-4). Muse expands it via `CODE_SIGN_ENTITLEMENTS_CONTENTS_TRANSFORMATION = expand-build-settings`.
- **PIPE_BUF (512B on Darwin, not 64KB) → RESOLVED: per-helper request pipe + central announce channel for discovery** (see Phase 1 transport change). One writer per data pipe dissolves the atomicity ceiling entirely; the tiny control messages ride a shared announce channel and stay under 512B. Keeps newline-JSON framing.
- **Timeouts are per-tool + declarative** in tool metadata (read ~10s; a future human-interactive tool can opt into 120s+). No per-call confirmation dialogs in v1, so the default holds everywhere for now.
- **`executeWithErrorHandling`** is EssentialMCP's own app-side wrapper (pattern to replicate), not a library API.
- **swift-sdk:** official `modelcontextprotocol/swift-sdk` 0.12.1+ (Package.swift is truth; the mcp-template README is stale).
- **The app KEEPS `com.apple.developer.contacts.notes`** (its own note UI needs it), so "drop the entitlement" is NOT an available defense — the wire boundary is the only line, hence INV-3's adversarial test.
