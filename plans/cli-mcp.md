## Architecture (converged with the three planning-consult agents; approved by two)

Two-process **relay**. The running **Mac Catalyst app** is the HOST: it owns TCC grants (EventKit/Contacts), the single `GuessWhoSync`/`ContactsRepository` instance, the iCloud sidecar, and all human-confirmation UI. A thin **relay** binary — bundled at `GuessWho.app/Contents/MacOS/`, optionally symlinked to `/usr/local/bin/guesswho` — is BOTH front doors.

> **⚠️ NAMING (execution blocker fixed): the bundled binary's `PRODUCT_NAME` is `guesswho-cli`, NOT `guesswho`.** The app target is `GuessWho` (`PRODUCT_NAME = $(TARGET_NAME)`), so its own executable is `GuessWho.app/Contents/MacOS/GuessWho` — and default **APFS is case-INsensitive**, so a helper named `guesswho` is the SAME directory entry as `GuessWho`: the Copy Files phase would clobber the app's Mach-O, and `Bundle.main.url(forAuxiliaryExecutable: "guesswho")` would resolve to the APP binary. So the embedded helper is `Contents/MacOS/guesswho-cli`. The **user-facing command stays `guesswho`** via the `/usr/local/bin/guesswho` symlink (→ `…/Contents/MacOS/guesswho-cli`). Everywhere below, the *bundled path* is `guesswho-cli` and the *typed command* is `guesswho`.

- `guesswho-cli run` (typed as `guesswho run` via the symlink) → long-lived **MCP stdio server** (for Claude Desktop / Cursor / Claude Code).
- `guesswho-cli contacts list …` → one-shot **human CLI**.

Both send the same typed requests over **App-Group named pipes** to the app; the app runs the logic and answers. The relay **does not link `GuessWhoSync`** — only the shared wire module + transport package.

```
Claude Desktop / terminal
   │ MCP JSON-RPC over stdio   |  ArgumentParser CLI flags
   ▼
 guesswho-cli (relay; bundled in .app; NOT linked against GuessWhoSync)
   │ newline-delimited JSON over named FIFOs (App Group container)
   ▼
 GuessWho.app (host — owns TCC, the single GuessWhoSync, the sidecar, confirm UI)
   │ direct calls into the SAME ContactsRepository/SyncService the UI uses
   ▼
 EventKit / Contacts + iCloud JSON sidecar
```

### Why relay, not a direct-to-Sync standalone CLI
1. **TCC ownership** — EventKit/Contacts auth is per-process and needs UI to grant; a client-spawned headless relay can't cleanly hold it. The running app already has it.
2. **Single writer** — `GuessWhoSync`'s per-key `NSLock` tables + contact-change watcher are per-process. Sidecar *files* are `NSFileCoordinator`-safe (no corruption), but two *processes* could interleave read-modify-write and bypass the in-memory locks — so routing all MCP writes through the app's ONE instance is necessary. **It is NOT sufficient on its own, though (data-integrity review):** the *identity-mint* path (`resolveOrMintGuessWhoID`) deliberately does NOT serialize across its `await`, and MCP adds a second unattended first-writer alongside the UI — see the CONCURRENCY & IDENTITY INTEGRITY bounds in Phase 2 (per-`localID` single-flight, post-mint verify, event Option B).

### v1 SCOPE STATEMENT (prevents scope creep — write this into the plan verbatim)
**v1 mutates GuessWho *sidecar* data only; system-store (EventKit / Contacts) *content* is READ-ONLY.** Reading system contacts/events is in; writing them is out of v1. Contact/event content writes are a *different coordination domain* (EventKit/Contacts, not the relay's single-writer sidecar guarantee), they need TCC *write* grants (a strictly harder grant than read with no clean mid-flight rollback), and they'd entangle the legitimate write path with the still-pending `com.apple.developer.contacts.notes` entitlement — so they get their own design pass, not a v1 phase.

### Load-bearing invariants (acceptance criteria, not conventions)
- **INV-1 Single writer by construction:** the relay target does NOT link `GuessWhoSync`. Enforced by the link line, verified in the build.
- **INV-2 Writes are visible everywhere:** every write tool routes through the SAME `ContactsRepository`/`SyncService` entry points the UI uses, so the change-watcher, iCloud push, and UI observation all fire. Test: an MCP-driven write is visible in the running app's UI and pushed to iCloud **without relaunch**.
- **INV-2b Host uses the LIVE app instance (read-side mirror of INV-1):** the host holds a reference to the app's existing `ContactsRepository`/`SyncService` (injected at app launch) and NEVER constructs its own store. **Enforcement = ARCHITECTURE CHECK, not a unit gate** (corrected after review): `ContactsRepository` is a *write-through read-model cache* (`ContactsRepository.swift:26` "not a second source of truth") — writes persist straight through, so there is NO tool-visible "un-persisted in-memory delta" to observe, and a naive positive-consequence test has no nameable state. Instead: a **link-layer / grep check** that the relay + host module never call a `ContactsRepository(...)` or `GuessWhoSync(...)` constructor (the instance is injected). **INV-2 is the behavioral proof** that the shared-instance wiring is correct (an MCP write shows up in the live UI + iCloud without relaunch — the consequence we actually care about, in the testable direction).
- **INV-3 Apple note NEVER crosses the wire (either direction):** the wire contact DTO has **no** `note`/`notes` field; the host mapper never reads `Contact.note` (Apple); no write tool accepts a note-shaped param; **search never MATCHES the Apple note** (a search that matches note text but returns note-less DTOs still leaks note content via *match-presence* — an agent binary-searches queries to reconstruct the note; this is a distinct assertion from "note absent from output"). The **allowed** "note" on the wire is GuessWho's own sidecar `ContactNote` (via `notes(at:)`), a *different type*. Pin the host search to `Contact.matches(searchQuery:)` (verified note-free, `Contact+Search.swift:63`) so the invariant doesn't rest on an unwritten implementation choice. **Tests (adversarial, structural):**
  - *Output absence:* a fixture whose Apple `Contact.note` holds a unique **sentinel** — the sentinel appears in **zero** tool outputs across read/list/search/write-echo **AND error responses** (force a failing call against the fixture, scan the error JSON); assert the key is **absent** from the encoded JSON (not merely empty).
  - *Search oracle:* `contacts.search(query: <note-sentinel>)` returns **ZERO** results (and for substrings of the sentinel) — proves match-presence can't leak note content.
- **INV-3b Wire DTOs are a POSITIVE FIELD ALLOWLIST, not a blocklist (strengthened after security review):** a blocklist naming only the Apple note + `guesswho://` keeps leaking internal fields as the model grows. Instead, each wire DTO **explicitly lists the fields it exposes**; anything not on the list never crosses. Concretely, the following MUST be dropped/absent from every DTO: the Apple `note`; `modifiedBy`/`deviceID` (a stable per-install UUID on ContactNote/Link/SidecarField/SidecarCell — `SidecarField.swift:16` etc., set from the UserDefaults device id `SyncService.swift:905-914` — an agent could correlate every record the user ever wrote); any raw `guessWhoID`/`localID`/`SidecarKey.id`; and the `guesswho://` identity URL. URL addresses source from `Contact.userVisibleURLAddresses` (`Contact.swift:177`), never raw `urlAddresses`. Since `Contact.matches(searchQuery:)` iterates **raw** `urlAddresses` (`Contact+Search.swift:86`), the wire search path must pre-filter identity URLs (pinning search to `matches()` does NOT close this). **Tests:** `contacts.search(query: "guesswho")` returns ZERO matches; no bare canonical-UUID (guesswho-pattern), no `localID`, no `modifiedBy`/device-UUID, and no `guesswho://` appears in ANY tool output; a golden-DTO test asserts each DTO's key set equals its declared allowlist.
- **INV-4 App + helper derive the app-group id from ONE shared per-channel build var (never a hardcoded string):** ✓VERIFIED in Muse — the group id is team-prefixed AND channel-suffixed (`group.<TEAM>.<parent-bundle-id>`), so the *string differs per channel*. **THREE distinct id derivations** (App Store, Setapp, Debug), across four nominal channels (TestFlight rides the App Store id). Debug matters: the **CLI app group carries its own `.debug` suffix** so a Debug helper can never reach a production app's CLI container (and vice-versa) — the per-channel derivation MUST include Debug or Debug builds silently mismatch (app + helper resolve to different containers). (NOTE: this is a dedicated CLI group; the app's OWN storage/bundle ids are NO LONGER `.debug`-suffixed in Debug — commit `622671a` de-suffixed them to production; the earlier "GuessWho already uses `.debug`-suffixed ids" premise in [[project_debug_build_identifiers]] is stale. The CLI group's own suffix is still the right call regardless, and it works because both targets expand the group from the same build variable within a build.) A hardcoded constant, or two vars that drift, = silent FIFO failure. INV-4 is the *static* guarantee; **Phase 0 exit criterion 3 (app + CLI resolve the SAME container path at runtime, on each channel's build INCLUDING Debug) is its runtime verification** — named together on purpose.
- **INV-5 Mac (Catalyst) only for v1:** pipe/wire code gated Mac-only so iOS builds don't pull in FIFO code with no host. Concretely: the relay binary is built + Copy-Files-embedded ONLY on the Catalyst-capable config; **an iOS-only build simply OMITS the Copy Files phase** (it must NOT try to embed a macOS Mach-O into an iOS `.app` — that would fail to codesign). Make the iOS build's absence-of-relay explicit so nobody wonders where it went.

### Consent & safety model (NO per-call confirmation dialogs)
The MCP client is headless — the human is in Claude Desktop / Terminal, not staring at GuessWho — so an in-app modal on an MCP write is a dialog **nobody is watching** (it blocks/times-out the call or forces alt-tab per write). Therefore: **consent is the read-only toggle, granted once in Preferences** (`isMCPReadOnly` / `isCLIReadOnly` — writes OFF by default). GuessWho deletes are **soft-delete/tombstone** at the package level, but per the security review that is only *safe for the user* once a **user-reachable recently-deleted/undo surface exists** — a v1 prerequisite for enabling writes (see Phase 2). This also removes the "longer interactive timeout" complexity. (Note: the inherited `ResponseManager` default is **5s**, `ResponseManager.swift:110` — not 10s; GuessWho sets timeouts **per-tool in metadata** regardless, so the exact inherited default is moot.) If Adam wants human-in-the-loop approval for agent writes, design it out-of-band (a notification the user acts on), NOT a modal blocking the tool call — flag as a product decision.

### Threat model (stated explicitly — from the adversarial security review)
The security posture rests on decisions that must be *stated*, not silently assumed:
- **The AGENT may be adversarial (prompt-injected).** An MCP agent can be steered by untrusted content it reads (a malicious calendar invite, a contact note authored by someone else, a web page). So the tool surface is treated as reachable by a confused/hostile caller: no tool exposes data the DTO allowlist doesn't name; wire IDs are unguessable sealed handles (no enumeration); writes are gated, rate-limited, audited, and recoverable; requests are size-capped.
- **The local machine / user account is TRUSTED (stated decision, not an omission).** The relay↔app channel is unauthenticated newline-JSON over App-Group FIFOs; any local process that can reach the group container could, in principle, send `initialize` + tool calls. v1 assumes a **single-user, single-account machine where local processes are trusted** — the same assumption the copy-path install already makes. **`helperId` MUST be an unguessable RANDOM token** (never sequential/pid-derived) so cross-helper response isolation can't be defeated by guessing. If the trust assumption is ever too weak (shared/multi-account Mac), the mitigation is a **per-session shared secret** established out-of-band along the install path, with the host rejecting unauthenticated peers — noted as a future hardening, not v1.

  - The wire ID is an **opaque, server-minted, UNGUESSABLE random handle** (a capability). The host keeps an **in-memory `[handle: ContactID]` map**; it resolves a handle back to a real `ContactID` via the sanctioned round-trip `ContactID.restorationToken` → `ContactRestorationToken` → `ContactsRepository.contact(restorationToken:)` (`ContactsRepository.swift:333`) — all HOST-SIDE. **Never serialize `ContactID`, `ContactRestorationToken`, a bare `guessWhoID`/`localID`, or any `SidecarKey.id` onto the wire.** (Do NOT add public accessors to `ContactID`.)
  - **Handle LIFETIME (from review — interacts with the reconnect state machine):** handles live for **one HOST RUN** and survive a **helper re-handshake WITHIN that run** (a helper reconnecting to the still-running app keeps using its cached handles). A **host restart rebuilds the in-memory map**, so handles minted before it become invalid — do NOT persist the handle map. Any unresolvable handle (host restarted, or the record was deleted) returns a typed **`staleHandle`** error with guidance to re-run the search/list — NEVER an opaque `notFound` that reads as a transport bug.
  - **Relationships:** a `Link`'s far endpoint is a `SidecarKey` carrying a raw canonical UUID (`Link.swift:10-12`) — project each endpoint to the **same sealed handle**, never the raw `SidecarKey.id`, so an agent can't harvest the contact-to-contact graph or resolve/enumerate contacts it wasn't handed.
  - **Events:** expose the **stable sidecar UUID when adopted, else an `eventKitID`-derived opaque handle** — both via the same sealed-handle scheme (no raw UUID on the wire). Reads do **NOT mint sidecars** (match `fields(for:)` no-mint; adoption stays a UI path). **Decision: `events.list` DOES surface un-adopted EventKit-only events** (real events the user sees), each as an `eventKitID`-derived handle; the MCP read never triggers adoption.
  - **Event-tag WRITE against an un-adopted event handle:** `SyncService.addEventTag(forEventUUID:)` needs a sidecar UUID, but an un-adopted event handle has only an `eventKitID`. **DECIDED (Phase 2, Option B): the write returns a typed "open the event in the app first" error and mints NOTHING** — mint-on-write is NOT built in v1 (it would race the UI adopt-on-load; see the Phase 2 event-tag bullet). A Phase 1 implementer should NOT build mint-on-write plumbing. (Reads never mint either.)
